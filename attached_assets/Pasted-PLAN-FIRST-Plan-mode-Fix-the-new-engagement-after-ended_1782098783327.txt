PLAN FIRST — Plan mode. Fix the "new engagement after ended" flow for
shadow teachers, and add a 3-month matching-fee waiver for re-requests.
Currently: the ended-engagement CTA links to /services with no clear
path to actually request a new shadow teacher. This fix makes the CTA
open a new match request form directly, with the fee waived if the
parent paid within the last 3 months for this same child.

═══ THE FLOW ═══

1. Parent's engagement ends → they see "Start a new engagement" CTA
   (already built). Clicking it now navigates to the new match request
   form directly (not /services), pre-filled with the previous match's
   data for this child.

2. Before showing the form, the backend checks whether this parent has
   a non-refunded ₹500 matching payment for this specific child within
   the last 3 months (90 days), anchored to the PAYMENT DATE (not the
   engagement end date).

3. If waiver applies (payment within 90 days, non-refunded):
   - Show the pre-filled form with a green banner: "Your matching fee
     is waived — you paid within the last 3 months for [child name].
     Review your details and confirm to get matched."
   - Submitting the form creates a new match request WITHOUT a Razorpay
     order. No payment step. Goes straight to matching.

4. If waiver does NOT apply (>90 days or no prior payment or refunded):
   - Show the pre-filled form with the normal ₹500 CTA: "Review your
     details and pay ₹500 to get matched."
   - Normal payment flow as today.

5. After submission (waived or paid), the new match request is created
   and the parent sees the candidate list as normal.

═══ WAIVER CHECK LOGIC ═══
Query: does a row exist in the payments table where:
- userId = current parent
- plan = "shadow_teacher_match" (or whatever the ₹500 matching plan
  is named — confirm the exact plan value)
- status = "paid" (or equivalent confirmed status)
- professionalId or childId scoped to THIS child (confirm which
  column ties the payment to a child — if payments doesn't have
  childId, check shadow_teacher_matches for the link)
- createdAt >= NOW() - INTERVAL '90 days'
- NOT refunded (status != "refunded")

If the payments table doesn't have a direct childId, join via
shadow_teacher_matches: payments.razorpayOrderId →
shadow_teacher_matches where childId = requested child.

New backend endpoint: GET /api/shadow-teacher/re-request-eligibility
?childId=X
- Auth: parent only
- Returns: { waived: boolean, previousMatch?: { city, conditions,
  languages, budget, goals, preferredModes, extraNotes } }
- waived: true if non-refunded ₹500 payment within 90 days for this
  child
- previousMatch: the most recent match request data for this child
  (for pre-filling the form) — always returned regardless of waiver
  status, since the form always pre-fills

The existing POST /api/shadow-teacher/request endpoint:
- Add a server-side waiver check BEFORE creating the Razorpay order.
  If waived=true (verified server-side, not just from frontend), skip
  order creation and create the match request directly with
  paymentId = "waived_rematch" (or a clear marker).
- Do NOT trust the frontend to declare the waiver — always verify
  server-side. The frontend passes childId; the backend independently
  checks the 90-day window.
- If not waived, normal flow (Razorpay order → verify → create match).
